## Overview
Learning plan generation transforms prioritized skill gaps into persisted `learning_plans` and `learning_plan_items` with week-wise roadmap projections and progress tracking. It supports both quick list views and structured week packing with subtopics/sub-subtopics.

## Input
Inputs include:
- Gap analysis output (`GapResult`)
- User + target role context
- Planning constraints (`daily_hours`, `deadline_weeks`, `hours_per_week`)
- Optional roadmap regeneration flag (`force_regenerate`)
- Assessment feedback inputs (`skill_name`, `score`, `failed_areas`, `passed_areas`) for remediation updates

API entry points:
- `GET /api/v1/users/me/learning-plan`
- `GET /api/v1/users/me/learning-roadmap`
- `PATCH /api/v1/users/me/learning-plan/{plan_id}/items/{item_id}/progress`
- `POST /api/v1/users/me/learning-plan/{plan_id}/items/{item_id}/assessment-feedback`
- `GET /api/v1/users/me/learning-plan/progress`

## Output
Plan output (`LearningPlanResponse`): flat ordered skill items with estimated hours/resources and optional `skill_rationale`.
Roadmap output (`LearningRoadmapResponse`): week-wise hierarchy with per-skill subtopics/resources, item IDs, item statuses, optional `skill_rationale`, and `estimated_total_weeks` for direct progress updates.

## Core Algorithm / Logic
1. Build ordered candidates.
`build_learning_plan` sorts gaps by dependency presence, priority score, importance, and mandatory flags, then applies optional hour budget from daily-hours/deadline.

2. Generate skill breakdown.
For roadmap mode, each skill is expanded by Gemini into 3-6 subtopics and 2-5 sub-subtopics, with strict total-hour normalization to gap-estimated hours.

3. Catalog-first resource selection.
Resource links/titles are sourced from the static catalog via `get_resources_for_skill` (not generated by Gemini on the happy path).
Current limits are constant-driven in the generator:
- `CATALOG_LIMIT_PER_FORMAT = 2`
- `MAX_CATALOG_RESOURCES_PER_SKILL = 4`

Selection order is deterministic: video/course bucket first, then article/doc bucket, capped at max-per-skill.
Resource entries use the strict `video|article|course|practice|docs` contract (`doc` normalizes to `docs`).
`estimated_hours` derives from `duration_minutes` when present; article/doc entries use a small positive default.

4. Gemini rationale (batch + cache).
Gemini is used for short skill-level rationale text (1-2 sentences) only, with no URL/title generation.
To reduce cost/latency, rationale generation uses:
- chunk-level batch prompt (`_gemini_skill_rationale_batch`)
- in-memory per-request cache keyed by `(skill_name, target_role, gap_type)`

5. Fallback structures.
If LLM breakdown fails, deterministic placeholders are generated (fundamentals/applied/advanced with core/practice children).

6. Dependency-aware leveling.
Skills are levelized using prerequisite graph logic (Kahn-style indegree processing with cycle escape heuristic).

7. Week packing.
Leveled skills are packed into weeks by `hours_per_week` without splitting a skill across weeks. A single oversized skill may occupy its own week. Prerequisites are respected so a skill is never scheduled before an unscheduled prerequisite.

8. Persistence and reload path.
Generated plans persist to `learning_plans` and `learning_plan_items` (including `skill_rationale`), plus normalized `learning_plan_item_resources` rows. Serialized subtopics/resources are also stored in `subtopics_json` as a wrapper object for compatibility.
Cached-load path deserializes both legacy subtopic-only payloads and the newer wrapper object, then repacks weeks for response.
Cached roadmap reload also restores persisted `priority_score` from `learning_plan_items` so reranked plans stay consistent after refresh.

9. Legacy-cache invalidation.
If persisted plan items exist without `subtopics_json`, roadmap endpoint regenerates once to migrate old cached plans to current shape.

10. Progress lifecycle.
Item progress supports `not_started`, `in_progress`, `completed`, and `needs_review`; completed items cannot be reverted. Plan status is derived from item aggregate completion. Completing a skill can rerank remaining incomplete items when prerequisite coverage changes.

11. Assessment feedback.
High-score assessments return `action_taken="upgraded_to_weak"`, mark the item as weak, and keep the existing breakdown. Low-score assessments return `action_taken="subtopics_regenerated"`, regenerate remediation subtopics from failed areas, preserve existing resources, and move the item to `needs_review`.

12. Item-tied assessments.
Assessment questions are tied to each `learning_plan_items` row (one question set per item instance), and submissions update item status directly (`completed` on pass, `needs_review` on fail).

## Catalog Fallback Rules
| Situation | Behavior |
|---|---|
| Catalog returns video + article | Use catalog rows only; no Gemini resource URLs/titles |
| Catalog returns only one bucket | Use partial catalog rows only |
| Catalog returns no video and no article | Log warning and fall back to `_gemini_resources` for that skill only |

## Key Files
backend/app/services/learning/path_generator.py: Plan/roadmap generation, LLM breakdown, packing, DB load/save logic.
backend/app/api/v1/endpoints/learning.py: Plan, roadmap, progress APIs and validation.
backend/app/schemas/learning.py: Typed contracts for weeks, skills, subtopics.
frontend/src/pages/RoadmapPage.jsx: Week-wise rendering and per-item status updates.

## Data Model
Reads/Writes:
- `learning_plans`
- `learning_plan_items` (`status`, `completed_at`, `priority_score`, `resource_type`, `skill_band`, `skill_rationale`, `subtopics_json`)
- `learning_plan_item_resources` (`title`, `provider`, `url`, `resource_type`, `estimated_hours`, `why`)
- role/user linkage through `users`, `roles`

## Edge Cases & Guards
Missing target role returns 400/404 depending on context. Progress API blocks completion reverts with explicit 400. Cached roadmap load handles malformed subtopics JSON with safe empty fallback. Single oversized skill can occupy its own week when it exceeds weekly budget.

## Current Limitations
LLM breakdown quality depends on external model output and may still require manual curation. Catalog coverage determines how often the system can avoid Gemini resource fallback for specific skills.

## Extension Points
Week 6: add an adaptive pacing feedback loop from actual completion velocity. Integrate external content providers into resource generation instead of placeholder entries.
